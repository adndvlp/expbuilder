     Hallazgos — Sistema de escalado actual vs Gorilla.sc

     Investigación de la codebase JsPsych Builder. Comparación entre el editor
     Konva (compile-time) y el runtime DynamicPlugin (jsPsych HTML) contra el
     algoritmo Gorilla documentado en gorilla-scaling-impl-plan (3).md.

     ---
     1. Algoritmo Gorilla (referencia)

     scaleX  = viewport_w / design_w
     scaleY  = viewport_h / design_h
     scale   = Math.min(scaleX, scaleY)
     xOffset = (viewport_w - design_w * scale) / 2
     yOffset = (viewport_h - design_h * scale) / 2
     screen_x = design_x * scale + xOffset
     screen_y = design_y * scale + yOffset

     Características clave:
     - Escala uniforme (mismo factor en X e Y).
     - Letterbox: márgenes simétricos cuando AR del viewport ≠ AR del diseño.
     - Math.min garantiza que el contenido nunca desborda.
     - Idéntico en runtime y editor (verificado con 7 capturas).

     3.1 Patrón CSS actual

     Ejemplo TextComponent.ts:285-291:
     left:      `calc(50% + ${xValue}vw)`   // xValue = jsPsych_x * 0.5 → [-50, 
     50]
     top:       `calc(50% - ${yValue}vh)`   // yValue = jsPsych_y * 0.5
     width:     `${widthVw}vw`
     height:    `${heightVh}vh` // (¡a veces vw, inconsistente!)
     transform: translate(-50%, -50%);
     position:  absolute;

     Container raíz (index.ts:154-161):
     position: fixed;
     inset: 0;
     width:  100vw;
     height: 100vh;
     overflow: auto;

     3.2 Problemas

     1. vw y vh independientes → escala no uniforme. Distorsiona el AR cuando
     viewport AR ≠ design AR.
       - Ejemplo: design 1000×1000, viewport 1920×1080:
           - Gorilla: scale = min(1.92, 1.08) = 1.08 (uniforme)
         - Runtime actual: 50vw ≈ 960px, 50vh ≈ 540px (factores distintos →
     cuadrado se vuelve rectángulo)
     2. Sin xOffset / yOffset: no hay letterbox. El "centro" siempre es 50%/50%
     del viewport, no del canvas escalado.
     3. Sin listener de resize: depende totalmente del re-flow CSS de vw/vh.
     Funciona pero no permite recalcular custom math.
     4. design_w/design_h no llegan al runtime como global. Solo están embebidos
      en las claves de screenLayouts (e.g., "1440x900"). El __canvasStyles se
     persiste en config (useConfigFromComponents.ts:170-175) pero no se lee en
     runtime.

     3.3 resolveScreenLayout (parche, no algoritmo)

     index.ts:100-134. Selecciona el screenLayouts[key] cuyo width-axis
     Manhattan distance al window.innerWidth sea < 200px. No aplica matemática
     de escalado — solo elige un override pre-guardado por preset.

     Si no encuentra match, no hace fallback de escalado: usa los valores tal
     cual (que ya están en formato vw%).

     3.4 Trial data en coordenadas de viewport

     index.ts:350-353 y 402-413:
     trialData[`${prefix}_coordinates`] = JSON.stringify({
       x: Math.round(window.innerWidth  * (0.5 + cx / 200)),
       y: Math.round(window.innerHeight * (0.5 - cy / 200)),
     });

     Coords grabados en px de viewport, no design-space. No comparables entre
     tamaños de pantalla.

     5. Componentes y su path de render

     Stimulus (puramente visuales, fáciles de escalar):
     - ImageComponent — <img> o canvas drawImage
     - TextComponent — <div> con cloze mode
     - HtmlComponent — HTML arbitrario
     - VideoComponent — <video>
     - AudioComponent — <audio> (no posicional)

     Response (con coords internas / hit-tests):
     - ButtonResponseComponent — botones, posicional pero pasivo (cosmético)
     - KeyboardResponseComponent — sin DOM (listener)
     - SliderResponseComponent — usa slider_width px + getBoundingClientRect
     - InputResponseComponent — <input> posicional
     - SketchpadComponent — canvas con getBoundingClientRect para puntero
     (SketchpadComponent.ts:507-533)
     - SurveyComponent — survey.js (mide clientWidth/offsetWidth internamente)
     - ClickResponseComponent — graba clientX/Y directo
     - AudioResponseComponent — recording widget
     - FileUploadResponseComponent — file input nativo

     Los response components con coords internas (Sketchpad, Click, Slider,
     Survey) son los que más sufren si se aplica transform: scale a un wrapper
     común — los getBoundingClientRect y clientX/Y quedan en escala viewport
     pero el bitmap interno del canvas / la geometría de Survey no.
